1. Overview

This application is a single-page document Q&A client. The browser lets users upload files, select which documents are active, ask questions, and receive answers from a backend service.

The page combines three layers:

  • HTML for structure: header, chat area, upload panel, document list, and buttons.
  • Tailwind CSS + custom CSS variables for layout, spacing, colors, animations, and responsive behavior.
  • JavaScript for state management, fetching data, opening a WebSocket, uploading files, polling answers, and updating the DOM.

2. Page Layout

Major regions

  • Main chat panel: messages area and input composer.
  • Sidebar: upload zone, processing queue, document selection list.

3. Tailwind + Custom CSS

The original page mixes Tailwind utility classes with a custom design-token layer built from CSS variables. That is a strong pattern when you want Tailwind for layout but still need runtime theme switching.

Tailwind handles: flexbox, spacing, width, typography sizing, breakpoints, rounded corners, visibility rules like lg:hidden and hidden lg:flex.

Custom CSS handles: theme variables, upload hover effects, custom checkboxes, scrollbar styling, animations, and icon state changes.

Key idea: set colors with CSS variables such as --bg, --card, and --accent; then let JavaScript switch the data-theme attribute on the root element.

4. JavaScript State Model

Core variables

  • documents: the current document list from the API.
  • allDocsEnabled: global toggle state.
  • processingQueue: uploads still being processed.
  • pendingQueries: queries waiting for completion.
  • API: base API path, set to /api.

5. REST API Flow

The page talks to a backend with fetch through a helper called safeFetch(). That function wraps network calls, reads the response as text first, attempts JSON parsing, and throws a clean error when parsing or HTTP status fails.

That design gives the UI one standard way to call the server.

GET /api/models
Loads model names into the select menu.
GET /api/documents
Loads indexed documents into the sidebar.
POST /api/upload
Uploads one file with FormData.
GET /api/upload/:id/status
Polls processing state.
POST /api/query
Starts a query job.
GET /api/query/:id/answer
Polls answer completion.

6. WebSocket Design

The page creates a WebSocket with either ws: or wss: depending on the current page protocol. That is important because secure pages should use secure sockets.

When the socket opens, the app changes the status dot to green, resets retry counters, and sends a lightweight ping every 30 seconds to keep the connection alive.

When the socket closes, the page uses exponential backoff to reconnect. That makes the app more resilient if the backend restarts.

function connectWS() { const proto = location.protocol === 'https:' ? 'wss:' : 'ws:'; ws = new WebSocket(proto + '//' + location.host + '/ws'); ws.onopen = () => { // mark connected // start ping interval }; ws.onclose = () => { // mark disconnected // reconnect with backoff }; ws.onmessage = (event) => { // parse message // update UI for query_answer or upload_complete }; }

7. Upload Flow

The upload area supports both clicking and drag-and-drop. The hidden file input is triggered when the upload panel is clicked. Drag events add and remove a visual dragover class.

Each file is uploaded in sequence. After upload, the returned document id is added to the processing queue. The UI then polls for status until the backend marks the document completed or failed.

Why this is good
  • Simple flow for beginners to follow.
  • Visual feedback during long processing.
  • Works even if WebSocket completion is missed, because polling is a backup.
Tradeoff
  • Repeated polling creates extra requests.
  • Sequential uploads are simpler, but slower for many files.

8. Chat Flow

The chat composer grows automatically as the user types, up to a max height. Pressing Enter sends the message; Shift+Enter creates a new line.

When the user sends a query, the UI:

  • adds the user bubble,
  • adds a loading bubble,
  • collects enabled document ids,
  • sends the query to the backend,
  • stores the query id in pendingQueries,
  • starts polling and also waits for a WebSocket response.

That means the app supports two answer paths: push via WebSocket and pull via polling.

9. DOM Rendering Patterns

The page uses JavaScript functions like addUser(), addBot(), addSys(), and renderDocList() to create HTML strings and inject them into the DOM.

The helper esc() safely escapes user text before rendering. That is a good start. The helper renderMd() adds simple markdown-like formatting for bold text and inline code.

Important teaching point: rendering HTML with string concatenation is fast to write, but it becomes harder to maintain as the UI grows. A future refactor could use template functions, web components, or a framework.

10. Responsive Design

The original page is mobile-aware. The sidebar becomes an off-canvas drawer below the large breakpoint. An overlay darkens the main content while the sidebar is open.

Tailwind classes such as sm:px-5, max-w-7xl, lg:hidden, and hidden lg:flex control what changes at different widths.

This is a practical example for new developers because it shows a real-world responsive pattern without using a framework.

11. Security Issues to Address

1. CDN script risk. The page loads Tailwind from a CDN. In production, pin versions, use Subresource Integrity where possible, or bundle locally.

2. Missing CSRF strategy. The upload, delete, and query endpoints are state-changing operations. If cookie auth is used, protect them with CSRF tokens or same-site cookie policy plus server-side validation.

3. File upload validation. The browser limits file types with accept, but that is not security. The server must validate MIME type, extension, file size, malware, archive bombs, and parsing limits.

4. WebSocket auth. The backend should authenticate the socket and authorize document access per user. Do not trust document ids from the client alone.

5. XSS concerns. The page escapes plain text, but any rich answer formatting from the backend must stay sanitized. Never inject raw model output as trusted HTML.

6. Access control. The client lets users choose enabled_doc_ids. The server must verify the user owns those documents.

7. CORS and origin checks. Limit allowed origins for both REST and WebSocket endpoints. Avoid open CORS policies.

8. Transport security. Use HTTPS and WSS in production. Without TLS, uploads and chat content can be intercepted.

9. Rate limiting. Add rate limits for uploads, query creation, delete actions, and WebSocket messages.

10. Logging hygiene. Do not log raw file content, secrets, tokens, or sensitive prompts. Use structured logs with redaction.

12. Suggested Improvements for a New Developer to Build Next

Move inline onclick handlers to addEventListener() for better separation of structure and behavior.
Extract the chat bubble HTML into reusable template functions.
Add a real markdown parser with sanitization for richer answer formatting.
Add cancel-upload and cancel-query support.
Use optimistic UI carefully for document toggles and deletes.
Split the code into modules: api.js, ws.js, ui.js, and state.js.
